# 33 输入法文章对照（产品/协议洞察）

来源：用户转发的豆包输入法逆向 →「33 输入法」产品文（课堂长录音动机、静态/动态逆向、三轮优化、上下文）。

与 `asr-quality-from-reverse.md`（Node/Swift/APK 技术对照）互补：本文更偏**产品观察 → 准确率根因**。

## 文章核心论点（与我们高度相关）

1. **时序完整性**：连接 / 发送 / 接收 / 麦克风 / UI **全部并发**；错误时序会直接掉准确率。
2. **三轮优化**（一次 PTT 内）：
   - 说话中：临时结果不断覆盖；
   - 短暂停顿：前句再优化（稳定结果）；
   - 松手后：「识别优化中」→ 最终文本可能与松手前最后一版不同。
3. **上下文才是准确率关键**：
   - **会话内**：当前应用、输入框类型、光标附近可发送文本；
   - **会话外**：历史输入/语音、联系人/常用词、标点策略、纠错积累。
4. 早期论坛复现「能出字但质量掉」——与我们/社区「裸 ASR」体验一致；缺的是端云协同纠错链路，不是单纯 Opus/WS。

## 建议优先级（个人学习工具视角）

**P0（体验/准确率杠杆大、实现面可控）**

1. UI 显式「识别优化中」：用户松手 → Finalizing，直到 `Closed(Succeeded)`；不要用松手前最后一帧 interim 当最终文案。
2. 区分并保留：最新 interim / 已稳定段 / 终态文本（对应文章三轮）。
3. 可配置 `sourceAppName`（至少别写死 Chrome）；若以后做悬浮入口，改成真实包名。

**P1（准确率上限，需继续动态逆向）**

1. 会话内上下文：光标前后文本、输入框类型（需对照真机抓包确认字段名）。
2. Node `NER_URL` / `x-api-resource-id: asr.user.context` 是否与「专有名词/联系人」相关——单独验证后再接。

**P2（产品形态，非协议）**

1. 课堂长录音模式（连续会话、归档）≠ 当前一键 PTT；可另开产品切片。
2. 个人词库 / 纠错记忆本地落盘。

## 文章对方法的提醒（值得保留）

- 先做**细致产品观察**再逆向，比「模糊一句让 AI 去挖」有效得多。
- 静态给「可能是什么」，动态请求博弈给「确实是什么」。
- 完整引擎不公开、只公开框架——合规与失效风险与我们 `provider-doubao-boundary` 一致。

## 结论

**有参考价值，而且补上了我们技术对照文档里偏弱的「为什么官方差很多」叙事：**  
差不只在 mic/VAD/尾音，更在 **三轮优化语义 + 会话/个性化上下文**。  
0.5.1 已把「时序与切段」拉近；下一刀应打在「松手后优化态 UI」和「上下文注入（先动态验证字段）」。

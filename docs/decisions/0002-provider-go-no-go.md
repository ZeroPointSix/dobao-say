# 决策记录 0002：ASR Provider Go/No-Go

- 状态：阻塞草案
- 日期：2026-07-19
- 关联任务：Linear ZER-102、ZER-103、ZER-104、ZER-105、ZER-106、ZER-107、ZER-108
- 决策范围：M0 可行性与合规门槛

## 背景

项目需要在进入 M1 前明确 ASR Provider 的实现和验证边界。当前候选私有 Provider 缺少已确认的许可证、服务条款许可、书面授权和可审计的官方接入资料，因此不能把技术上可能实现等同于当前有权实施。

本记录是工程风险决策，不替代法务意见。授权、条款、产品用途或数据处理方式发生变化时，必须重新评审并留下可追溯证据。

## 当前结论

### 当前私有 Provider：NO-GO

在许可证、服务条款和书面授权未确认前，停止以下工作：

- 实现、移植或集成私有 Provider；
- 使用真实凭据、真实设备身份或真实服务进行网络验证；
- 从非官方实现、客户端产物或流量记录复刻协议；
- 将私有 Provider 作为 MVP、生产或分发版本的依赖；
- 用一次成功演示替代稳定性、安全性和合规证据。

该结论阻塞 ZER-106、ZER-107 中依赖真实私有服务的验收项。两项任务目前仍保持原始 Backlog 描述，尚未同步为受限范围；在任务描述、状态和验收标准完成同步前，不得启动。

### 通用离线与 Fake Provider：GO

可以继续推进：

- provider-neutral 的 ASR 契约、会话状态机和错误模型；
- 使用团队自行定义事件的 Fake Provider 与本地回环服务；
- 使用合成 PCM 的音频格式、分帧、背压和资源释放测试；
- 超时、取消、断网模拟、final 去重和幂等关闭测试；
- 通用凭据接口、Android Keystore 封装及脱敏测试；
- 不连接真实私有服务的 Android 录音生命周期验证。

这些成果不得隐含任何特定私有 Provider 的兼容性、授权状态或可用性结论。

### 官方或书面授权 Provider：CONDITIONAL GO

只有在下列条件全部满足后，才可以建立独立的 Provider 可行性评估：

1. 服务方或权利人提供可核验的书面使用与测试授权；
2. 法务确认适用许可证、服务条款、隐私、数据处理和分发边界；
3. 获得官方或经授权的版本化接入文档；
4. 使用合法签发的沙箱、测试租户和最小权限短期凭据；
5. 明确速率限制、数据保留、删除、地域和跨境处理规则；
6. 测试参与者已知情同意，测试设备和账户归团队合法控制；
7. 评估实现与通用核心分包隔离，能够停用和删除；
8. 安全审查确认凭据、音频和转写内容不会进入普通日志或提交物。

CONDITIONAL GO 只允许在单独任务、隔离分支或隔离仓库中做可行性评估，不代表自动进入 MVP，也不得自动解除 M1。评估结果必须重新提交 ZER-108 审批；任一条件缺失时维持 NO-GO。

## Clean-room 边界

允许进入正式代码和测试的内容：

- 团队独立定义的 provider-neutral 接口、状态和错误；
- 公开标准及其公开测试向量；
- 合成音频、人工文本和团队自定义 Fake 事件；
- 官方文档或书面授权资料允许使用的最小必要内容。

实现人员不得把未授权材料直接或间接转换为产品代码。外部材料观察记录必须与正式实现隔离；正式实现只能依据独立规格或明确授权资料进行。

## 当前隔离违规

正式公开产品仓库当前存在独立的 `reverse-sample` 研究历史及其公开派生分支，承载未经批准的第三方研究材料。它与 `main` 没有共同祖先，但仍属于同一个公开仓库的可访问引用。

这构成 ZER-103 所要求“参考研究与正式产品角色明确隔离”的当前违规：

- 不在默认分支不等于隔离；
- Draft PR 不会阻止公开分支被读取；
- 关闭某个 PR 不会移除基础分支、派生分支、历史或远端大文件对象；
- 这些材料不得作为产品代码、PR #1、PR #3 或任何正式测试的输入。

在来源、授权、访问范围、保留要求和清理方案由仓库所有者、法务与安全负责人共同确认前：

- 暂停继续上传、分析、合并、构建或分发相关材料；
- 不在公开 Issue、PR 或文档中追加材料细节；
- 不执行未经批准的删除或历史改写；
- ZER-103 的 clean-room 隔离验收不得通过。

## 禁止进入代码、测试和提交物的材料

- 未授权参考源码及其翻译、改写或结构性复刻；
- 私有端点、协议路径、请求头、消息名称、字段和枚举；
- 私有设备身份、客户端指纹、注册或凭据获取流程；
- 私有签名、密钥派生、票据或封包参数；
- 抓包载荷、客户端提取物和反编译产物；
- 真实 Token、Cookie、密钥、设备凭据或账户资料；
- 未脱敏的真实音频、用户转写文本和服务响应；
- 能够还原上述内容的日志、fixture、截图或实验报告。

## 责任与后续任务机制

需要明确并记录以下责任：

- 产品负责人：维护 ZER-102、支持范围、MVP 边界和替代 Provider 决策；
- 技术负责人：确认 provider-neutral 核心、隔离架构、回滚与删除能力；
- Android 负责人：提供设备矩阵、权限和真机系统证据；
- 安全负责人：负责秘密、日志、凭据、公开暴露和隔离处置审查；
- 法务/隐私负责人：确认授权、条款、数据处理和分发条件；
- Provider 评估负责人：仅在获批后执行独立可行性评估。

尚未存在的后续工作必须创建独立 Issue，至少包括：

1. 研究材料隔离与公开暴露处置；
2. Provider 授权、服务条款与隐私审查；
3. 获授权 Provider 的隔离可行性评估；
4. ZER-106 通用凭据安全范围拆分；
5. ZER-107 离线音频与 Fake/回环范围拆分。

这些 Issue 必须写明负责人、证据链接、完成条件和禁止事项。不能用 ZER-108 的一条评论替代执行任务。

## 复审与批准机制

ZER-108 的状态变化必须遵循：

1. 每个证据矩阵条目链接到仓库、测试记录、Linear Issue 或经批准的外部文件；
2. 产品、工程、安全、法务/隐私四类责任人分别给出明确批准或阻塞意见；
3. 所有 CONDITIONAL GO 条件必须有负责人、后续 Issue 和截止/复审触发条件；
4. 结论同步到 ZER-102 和 Notion 产品规格，并在 ZER-108 留下链接；
5. 当证据足以支持“当前路径不可继续”，且四类责任人共同批准该风险结论时，可以正式接受 NO-GO；接受 NO-GO 不要求先消除导致 NO-GO 的阻塞事实；
6. GO 只有在其适用门槛全部满足并获得四类批准后才能正式接受；
7. CONDITIONAL GO 可以在条件、负责人、后续 Issue 和复审机制明确后作为决策被接受，但在条件未满足时仍保持 M1 阻塞；
8. 只有已接受的 GO，或条件全部满足并再次获得四类批准的 CONDITIONAL GO，才能解除 M1；
9. 任何新 Provider、用途扩大、条款变化或数据处理变化都必须重新进入本流程。

当前批准状态：未完成。PR #3 在独立审查通过后可以作为临时工程门槛合入；合入只固化当前治理边界，不代表 ZER-108 已完成，也不解除 M1。复审期间 PR 保持 Draft。

## 影响

- ZER-103：未经批准的第三方研究材料历史导致隔离验收阻塞。
- ZER-106：不得启动原始私有注册、握手或凭据任务；先同步范围，再拆出通用凭据安全工作。
- ZER-107：不得启动原始真实流式链路任务；先同步范围，再拆出离线音频和 Fake/回环工作。
- ZER-108：负责维护证据矩阵并组织跨职能批准。
- ZER-102 与 Notion：当前尚未同步，完成同步并提供链接前不得解除 M1。
- 官方或书面授权 Provider：仅可先做隔离可行性评估，不自动纳入 MVP。
- UI 和其他 M1 功能不得通过引入未授权 Provider 绕过本门槛。

## 复审触发条件

出现以下任一变化时重新评审本记录：

- 获得新的官方 Provider、书面授权或适用条款；
- Provider 的数据处理、技术接口或分发政策变化；
- 研究材料的可见性、保留或隔离状态变化；
- 产品从内部验证扩展到外部测试、发布或商业用途；
- 安全、隐私或法务发现新的重大风险。
